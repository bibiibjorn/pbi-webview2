# pbi-webview2 v1.2.0 Feature Buildout — Implementation Plan

> **For agentic workers:** Use superpowers:subagent-driven-development or executing-plans to implement task-by-task. Steps use `- [ ]` checkboxes.

**Goal:** Add the empirically-validated feature catalog from the two 2026-07-15 research reports — model/DAX introspection, universal visual-data extraction, render-stable waits, editor-buffer safety, page digest, richer interaction, and a cross-cutting robustness layer — while excluding the dead-ends proven unreachable over CDP.

**Architecture:** Same as today — `server.js` registers tools from `src/tools.js`; DOM logic lives as self-contained page functions in `src/pagefns.js` (serialized to the WebView, no module refs); Monaco/DAX work runs in the `daxQueryView` CDP target; raw CDP sessions (`page.context().newCDPSession`) for `Performance.getMetrics`. Every tool returns `{ok:...}`/`{connected:...}` JSON via `guard(()=>withReport(...))`.

**Tech Stack:** Node ≥20, `@modelcontextprotocol/sdk`, `playwright-core`, `zod`. Windows-only.

## Empirical grounding (verified live against R0105, Desktop 2.155 / Edg150, 2026-07-15)

These facts drive the design — do NOT re-derive; they were probed live:

1. **`INFO.VIEW.MEASURES/TABLES/COLUMNS/RELATIONSHIPS()` all work** inside `EVALUATE` in the daxQueryView target. MEASURES columns: `ID, Name, Table, Description, DataType, Expression, FormatString, IsHidden, State, KPIID, IsSimpleMeasure, DisplayFolder, DetailRowsDefinition, DataCategory, FormatStringDefinition, LineageTag`. Counts on R0105: 124 tables, 1042 cols, 112 rels, 823 measures. **Precondition: the DAX query view must be OPEN in Desktop** (the tools cannot open ribbon views) — same precondition as `pbi_read_dax_editor`.
2. **Chart data VALUES are NOT in the SVG DOM at rest.** Verified: donut arcs and column `rect`s carry `role="option"` but `aria-label="null"`; only category labels are present as `<text><title>`. So "read chart by scraping aria-labels" **does not work**. The universal extractor is **"Show as a table"** (right-click a DATA POINT — not whitespace — → menu item) or the **Alt+Shift+F11** accessible show-data table, then read the grid. Right-clicking visual whitespace gives only `["Group","Summarize"]`; the data-table item needs a data point.
3. **Render-stable signal:** `Performance.getMetrics` `LayoutCount`/`RecalcStyleCount` climb during a repaint then go FLAT when settled (measured: 0→102→105→flat@~1000ms on a page switch). A NEW CDP session resets counters to 0, so `wait_stable` must sample deltas WITHIN ONE cdp session. Secondary signal: `[aria-busy="true"]` count (0 when idle).
4. **Monaco `editor.action.formatDocument` IS a supported action** in the daxQueryView editor → `pbi_format_dax` is feasible via `editor.getAction('editor.action.formatDocument').run()`.
5. **`window.monaco` is present on reportView, daxQueryView, tmdlView, dataExploreView** (model counts 2/3/3/1). `modelView` exposes a `desktopModelView` webpack global (not used here).
6. **Warm `page.evaluate` = 1–3ms**; the only real costs are the ~8s cold attach and mid-render blocking. So batching many reads into ONE evaluate is the main speed lever.
7. **Filters pane**: containers `filterPane`/`drillthroughPane filterPane filterPaneClassic` exist, but filter-CARD DOM only populates when the pane is OPEN; on R0105 it was collapsed (0 cards found). Filters extraction is best-effort and must open the pane first.
8. **CDP Network cannot see DAX/AS queries** (native msmdsrv IPC) — no network-capture feature. **Dirty state unreachable** (native WPF shell) — already handled by intent guards. **Query plans/SE-FE unreachable** over CDP — route to XMLA tools.

## Excluded (dead-ends — do NOT build)
- `pbi_read_chart` by SVG aria-scrape (values absent) → replaced by `pbi_show_as_table`.
- Network query capture; query plan / SE-FE timings; field-well bindings over CDP; export-data-to-file (redundant w/ show-as-table); modelView diagram reading; any "fix dirty detection" attempt.

- **`test/smoke.mjs`** — update EXPECTED_TOOLS + exact count.

## WAVE 1 — Safety + Model/DAX (highest value, lowest DOM risk)

### Task 1: `pbi_editor_buffer` — capture/restore Monaco buffer (safety primitive, ship first)

**Files:** Modify `src/tools.js`; reuse `F.readMonacoText`/`F.setMonacoText` (exist).

- [ ] **Step 1:** Add a module-level `Map` in tools.js `const _editorBuffers = new Map();` to hold captured text keyed by target (`dax`|`tmdl`).
- [ ] **Step 2:** Register `pbi_editor_buffer` with `inputSchema: { action: z.enum(['capture','restore']), target: z.enum(['dax','tmdl']).optional() }` (default `dax`). Resolve the sibling page by url (`daxQueryView`/`tmdlView`) exactly as `pbi_dax_query` does.
- [ ] **Step 3:** `capture`: read `F.readMonacoText` from that page; store `_editorBuffers.set(target, text)`; return `{ok:true, captured:true, target, length}`. `restore`: if no stored buffer → `{ok:true, restored:false, reason:'nothing captured for '+target}`; else `F.setMonacoText` it back, return `{ok:true, restored:true, target}`.
- [ ] **Step 4:** Wire auto-capture/restore into `pbi_dax_query`: before overwriting, capture the current buffer to `_editorBuffers.set('dax', prior)`; add a `restore?:boolean` param (default false) — when true, after reading results, restore the prior text. Update `pbi_dax_query` description: remove "unrecoverable" wording, note `restore:true` re-instates prior editor text.
- [ ] **Step 5:** Smoke: add `pbi_editor_buffer` to EXPECTED_TOOLS; `npm test`.
- [ ] **Step 6:** Commit `feat: pbi_editor_buffer + dax_query restore (fixes unrecoverable-overwrite)`.

## WAVE 2 — Render-stable + robustness layer (harden before more DOM surface)

### Task 5: `src/render.js` — `waitStable` via Performance metrics

**Files:** Create `src/render.js`; modify `src/tools.js` (register `pbi_wait_stable`).

- [ ] **Step 1 (render.js):** Export `async function waitStable(page, { timeoutMs=15000, quietTicks=2, intervalMs=350 } = {})`. Open ONE cdp session (`page.context().newCDPSession(page)`), `Performance.enable`. Loop: read `LayoutCount`+`RecalcStyleCount`; also `page.evaluate(()=>({busy:document.querySelectorAll('[aria-busy="true"]').length, vis:document.querySelectorAll('.visualContainer').length}))`. Stable when Layout+Recalc unchanged for `quietTicks` consecutive samples AND busy===0. Detach cdp in a `finally`. Return `{stable, elapsedMs, ticks, lastLayoutCount, visibleVisualCount, busy}`. NB counters reset per-session so only DELTAS across THIS loop matter.
- [ ] **Step 2 (tools.js):** Register `pbi_wait_stable { timeoutMs?, quietTicks? }` → `guard(()=>withReport(p=>waitStable(p, args)))` returning `{connected:true, ...}`.
- [ ] **Step 3:** Refactor `pbi_goto_page` and `pbi_reload` to OPTIONALLY use `waitStable` instead of the fixed settle (add `stable?:boolean` param; when true, after nav call `waitStable`). Keep the existing default behavior for back-compat.
- [ ] **Step 4:** Smoke + commit `feat: pbi_wait_stable (render-settle via Performance.getMetrics); goto_page/reload stable opt-in`.

### Task 6: Robustness — `resolveByRole` resolver + error taxonomy

**Files:** `src/pagefns.js` (`resolveByRole`); `src/tools.js` (shared `REASON` enum + use in new tools).

- [ ] **Step 1 (pagefns.js):** Add `resolveByRole({role, name, nameExact})` page fn: query `[role="${role}"]`, match accessible name (aria-label || textContent) exact-then-contains, tag with `data-pw="pw-role"`, return `{found, selector, matchedName, candidates}`. This is the version-drift-resilient primitive new tools prefer over CSS classes.
- [ ] **Step 2 (tools.js):** Add `const REASON = { NOT_FOUND:'not-found', NOT_READY:'not-ready', WRONG_VIEW:'wrong-view', CANVAS_BUSY:'canvas-busy', NOT_EXTRACTABLE:'not-extractable' };` and use these string codes in the `reason` field of ALL new tools (Wave 2+). Add a `code` field alongside human `reason` where a tool can fail multiple ways.
- [ ] **Step 3:** Commit `feat: resolveByRole resolver + structured reason codes`.
